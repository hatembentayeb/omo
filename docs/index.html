The omo Manual

Your entire infrastructure. One terminal. Keyboard-driven operations dashboard with 12 plugins, KeePass secrets, and a built-in package manager.

Keyboard-first Plugin architecture KeePass secrets Written in Go

What is omo?

omo (OhMyOps) is an operations dashboard that runs entirely in your terminal. Instead of juggling dozens of browser tabs, CLI tools, and dashboards, omo gives you a single keyboard-driven interface to manage your entire infrastructure stack.

                    omo
                     |
      +---------+----+----+---------+
      |         |         |         |
   Docker    Redis     Kafka    ArgoCD   ...12 plugins

One binary. One config directory. Every service you run.

The Problem

You have Redis in one terminal, docker ps in another, k9s somewhere, the ArgoCD UI in a browser, and your AWS cost dashboard in yet another tab. Context switching kills your flow.

The Solution

omo loads each service as a plugin and presents it through a consistent, keyboard-driven TUI. Filter, sort, search, and act on any resource — all without leaving your terminal.

  • Keyboard-first — every action is one or two keystrokes away
  • Plugin architecture — only load what you need, build your own in Go
  • Secrets management — KeePass (KDBX4) integration keeps credentials out of config files
  • Cross-platform — Linux, macOS, Windows (Linux recommended for plugin .so support)
  • Fast — written in Go, launches in milliseconds

Installation

One-line Install (Recommended)

bash
curl -fsSL https://raw.githubusercontent.com/hatembentayeb/omo/main/install.sh | bash

This downloads the latest release for your platform, installs the omo binary to /usr/local/bin, and creates the ~/.omo directory structure.

From Source

bash
git clone https://github.com/hatembentayeb/omo.git
cd omo
make all

Builds the omo binary and all plugins locally, then installs everything to ~/.omo/.

Custom Install Directory

bash
OMO_INSTALL_DIR=~/.local/bin curl -fsSL https://raw.githubusercontent.com/hatembentayeb/omo/main/install.sh | bash

Post-Installation

Launch omo

Run omo in your terminal. You'll be prompted for your KeePass password on first launch.

Open the Package Manager

Press p to open the built-in package manager.

Sync the Plugin Index

Press S to download the latest plugin index from the official repository.

Install Plugins

Press A to install all available plugins, or select individual ones.

Architecture

omo is structured as a thin host binary that loads plugins at runtime via Go's plugin package. Each plugin is compiled as a shared library (.so) and discovered from ~/.omo/plugins/.

Project Structure
cmd/omo/              Main binary — boots TUI, loads plugins
internal/
  host/               Plugin host, UI shell, navigation
  packagemanager/     Plugin install/update/remove
  registry/           Plugin discovery and manifest
pkg/
  pluginapi/          Public API that plugins compile against
  secrets/            KeePass KDBX4 secrets provider
  ui/                 Shared TUI components (tables, modals, filters)
plugins/
  redis/              Each plugin is a Go shared library (.so)
  docker/
  kafka/
  ...
dev/                  Docker Compose environments + seed scripts
config/               Default YAML configs installed to ~/.omo/configs/
index.yaml            Official plugin index (consumed by package manager)

Plugin Interface

Every plugin exports a single symbol:

Go
var OhmyopsPlugin pluginapi.PluginInfo

Where PluginInfo provides the plugin's name, version, description, and a CreateView function that returns a tview.Primitive. The host calls CreateView when the user selects the plugin from the sidebar.

KeePass & Secrets

omo uses a KeePass KDBX4 file as its single source of truth for all connection details and credentials. YAML config files only control which environments are enabled and UI settings — they never contain connection details like hosts, passwords, or tokens.

How It Works

Auto-Bootstrap

On first launch, omo creates a KeePass database at ~/.omo/secrets/omo.kdbx and a key file at ~/.omo/keys/omo.key. You'll be prompted to set a master password.

Store Connection Details

All connection details are stored in KeePass with the path convention: pluginName/environment/entryName. For example, redis/development/local stores the host, port, username, and password for your local Redis instance.

Configure Environments

YAML config files only define which environments are active and UI preferences — they never embed credentials:

~/.omo/configs/redis/redis.yaml
environments:
  - name: development
    enabled: true
  - name: production
    enabled: true

ui:
  refresh_interval: 5
  max_keys_display: 1000

Secret Entry Fields

FieldDescription
TitleDisplay name for the entry
UserNameUsername or access key
PasswordPassword or secret key
URLConnection URL or endpoint
NotesAdditional notes
Custom attributesArbitrary key-value pairs (e.g., certificates, private keys)

Plugin Secrets API

Plugins access secrets via pluginapi.Secrets(), which exposes:

MethodDescription
Get(path)Retrieve a secret entry by path
Put(path, entry)Create or update a secret
Delete(path)Remove a secret
List(prefix)List all paths under a prefix
Reload()Re-read the database from disk
Close()Flush changes and close
Note

The KeePass database and key file are protected with your master password. The key file uses KeePass v2.0 XML format with a 32-byte random key. Back up both files — if you lose them, your secrets cannot be recovered.

Using an Existing KeePass Database

If you already have a KeePass database, you can point omo to it by placing it at ~/.omo/secrets/omo.kdbx or creating a symlink. Just ensure it follows the pluginName/environment/entryName group structure so plugins can discover their secrets automatically.

Configuration

All configuration lives under ~/.omo/. Each plugin has its own YAML config file that controls environment groups and UI settings. Connection details belong exclusively in KeePass.

Example: Redis configuration
# ~/.omo/configs/redis/redis.yaml
environments:
  - name: development
    enabled: true
  - name: staging
    enabled: true
  - name: production
    enabled: true

ui:
  refresh_interval: 5
  max_keys_display: 1000
  enable_slowlog: true

You can edit configs directly with your editor, or press the config key inside omo to open the built-in editor for the current plugin.

KeePass-first model

Connection details (hosts, passwords, tokens, certificates) are never stored in YAML config files. They live exclusively in KeePass under pluginName/environment/entryName. YAML files only control which environments are enabled and UI behavior.

Tip

Default config templates are installed from the config/ directory in the repository when you first install a plugin.

Package Manager

omo includes a built-in package manager that fetches plugins from the official index. Press p from any screen to open it.

KeyAction
SSync plugin index from remote
AInstall all available plugins
EnterInstall / update selected plugin
DRemove selected plugin
UUpdate selected plugin

Plugins are downloaded as compressed archives, verified against SHA-256 checksums, and extracted to ~/.omo/plugins/.

Writing Your Own Plugin

omo's plugin system is designed to be simple. A plugin is just a Go package compiled as a shared library that exports a single OhmyopsPlugin variable.

Create the plugin directory

Add a new directory under plugins/yourplugin/.

Implement the interface

Create a main.go file that exports OhmyopsPlugin:

plugins/yourplugin/main.go
package main

import "omo/pkg/pluginapi"

var OhmyopsPlugin = pluginapi.PluginInfo{
    Name:        "yourplugin",
    Version:     "0.1.0",
    Description: "My custom plugin",
    Author:      "you",
    License:     "MIT",
    CreateView: func(app pluginapi.App) pluginapi.View {
        // Build and return your tview.Primitive here
    },
}

Add to the index

Add an entry to index.yaml with your plugin's metadata.

Build

Run make all to compile everything, or build just your plugin with go build -buildmode=plugin.

Note

Refer to any existing plugin under plugins/ for a complete, working example. The pkg/ui/ package provides shared TUI components (tables, modals, filters) that you can reuse.
